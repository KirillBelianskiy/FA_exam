# Правила подготовки экзаменационных ответов

## Цель

Ответ должен быть недлинным, но полным.
После одного прочтения должны быть понятны:

- определение;
- основная идея;
- механизм или алгоритм;
- причина корректности;
- сложность и ограничения, если они требуются вопросом.

Текст должен быть пригоден для устного ответа экзаменатору,
а не выглядеть как набор несвязанных тезисов.

## Точность

1. Не писать непроверенные сведения ради объёма.
2. Если точность утверждения не подтверждена, прямо писать:
   `не уверен`, `это упрощение` или `эту деталь не подтвердил`.
3. Если исходный ответ или пользовательская формулировка ошибочны,
   исправлять их и кратко объяснять слабое место.
4. Различать:
   - гарантию языка или стандарта;
   - математическое определение;
   - свойство конкретной реализации;
   - учебное упрощение.
5. Не скрывать расхождения между лекцией и более надёжным источником.

## Полнота без лишнего объёма

1. Сверять ответ со всеми подпунктами исходного экзаменационного вопроса.
2. Не добавлять справочные детали, которые не помогают ответить на билет.
3. Не повторять одну и ту же теорию в соседних темах без необходимости.
4. Для алгоритма обычно нужны:
   - входные данные и представление;
   - инвариант;
   - последовательность действий;
   - завершение и корректность;
   - оценка времени и памяти.
5. Для определения нужны существенные условия,
   без которых определение или следующая формула становятся неверными.

## Примеры и код

1. Пример должен непосредственно объяснять текст текущего вопроса.
2. Не использовать случайный код только ради наличия синтаксиса.
3. Для конспекта предпочтителен короткий фрагмент синтаксиса,
   а не полностью рабочая программа.
4. Не добавлять `using`, `Main`, вспомогательные классы,
   методы запуска и вызовы библиотек только ради компилируемости примера.
5. Фрагмент может быть схематичным и не обязан компилироваться отдельно.
   Пропущенные несущественные части нужно обозначать через `...`,
   комментарий или поясняющий текст.
6. Пример следует размещать сразу после определения,
   которое он иллюстрирует.
7. Нельзя использовать необъяснённые вспомогательные методы и типы,
   создающие впечатление, что они относятся к стандартной библиотеке.
8. Комментарии должны объяснять:
   - что делает конструкция;
   - зачем она используется;
   - с каким термином билета она связана.
9. Комментарий не должен просто повторять синтаксис строки.
10. После примера нужно словами сформулировать вывод.

## Порядок проверки

1. Сначала прочитать:
   - формулировки вопросов;
   - весь проверяемый ответ;
   - локальные правила;
   - релевантные лекции и исследования.
2. До изменения текста составить список:
   - фактических ошибок;
   - спорных формулировок;
   - неполных мест;
   - уже нормальных частей.
3. Сравнить, что необходимо добавить,
   а что является лишним и перегружает ответ.
4. Только после полного анализа составлять задания субагентам.
5. Субагенты сначала анализируют и не изменяют общий файл,
   если им явно не назначена отдельная область редактирования.
6. После сведения результатов внести одну согласованную серию правок.
7. Повторно проверить итоговую версию после редактирования.

## Субагенты

Каждый субагент должен получить:

- точный диапазон тем;
- путь к формулировкам вопросов;
- путь к локальным правилам;
- приоритет источников;
- запрет выдавать предположение за факт;
- требование прямо отмечать неуверенность;
- запрет изменять общий файл на этапе первичного анализа.

Для C# и .NET искать прежде всего:

- Microsoft Learn;
- спецификацию C#;
- ECMA-335 / CLI.

Для алгоритмов искать прежде всего:

- ITMO / NEERC Wiki;
- локальные лекции;
- первичные статьи и официальную документацию,
  если надёжной страницы ITMO нет.

Субагент обязан знать эти источники до начала проверки.

## Редактирование файлов

1. Не создавать альтернативный файл с ответами,
   если пользователь прямо этого не просил.
2. Исправлять существующий файл.
3. Сохранять уже корректные части и пользовательские изменения.
4. Не делать массовое расширение без доказанной необходимости.
5. После правок проверять:
   - количество вопросов и ответов;
   - структуру заголовков;
   - Markdown и формулы;
   - длинные строки;
   - внутренние противоречия;
   - связь примеров с текстом.

## Стиль

1. Не использовать ненужные комплименты и похвалу.
2. Не писать длинные сплошные абзацы.
3. Разделять смысловые шаги переносами строк.
4. Формулировать уверенно только подтверждённые положения.
5. Слабые места итогового ответа называть прямо.
